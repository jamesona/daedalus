Daedalus - Documentation

Play Area

The play area consists of a square table: 11 rows, with 11 cells. Accounting for walls, this results in a 9x9 space to explore in each room. The appearance of each cell is set by css, and controlled with data attributes. Items within a cell are considered "actors" and can include the player, a monster, or loot items on the floor. If an "actor" is provided, it will be rendered using an :after pseudo element, derived from data attributes. Rooms are randomly generated, with 2-4 exits to other rooms.

Menu

Menus are implemented as elements in a persistent containing element. The menu-container is positioned above the play area using z-index and absolute positioning. This has the effect of ensuring the menu is visible, and the play area can't be interacted with while the menu is open. Buttons are created by adding literal button or a tags to the menu element.

Data

Game data is kept in a model space, that can be saved, exported, and generated separately from the user interfaces and play area. The data model holds two structures, an instance of the world map and an instance of the player class. The map class holds data to represent all of the cell data for a room, including tile types, actors, and neighor rooms. The game class also contains pointers to libraries of possible creatures and items, ajax loaded from an external source.

Audio

Audio in the game is handled by an audio controller class, capable of adding and manipulating audio tags. Created tags are not appended to the document, and can only be manipulated through the element pointers saved in the controller.

Game Logic

Game logic currently consists entirely of a completed set of menu interactions, an event stack, a game tick loop, and a series of computations in the creature class that automatically poll current magical effects, abilities, and equipped items to determine derived stats and values using D&D style formulas. Logic that still needs to be implemented consists of player navigation, player interaction with other actors, and navigation to other rooms. Rudimentary player movement existed in an older version of the app, but has not yet been re-implemented in the current re-based version. Logic exists for sending and handling events between app components.
